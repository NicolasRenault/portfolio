Yes, absolutely! If you prefer keeping your HTML clean or plan to reuse this layout across different pages, moving those utility classes into a custom .wrapper class in your CSS file is a standard and highly recommended approach.
In Tailwind, you do this using the @apply directive.
Here are two ways to write it, depending on how much you want to rely on Tailwind's utility syntax versus standard CSS.
Option 1: The "All-in-One" @applyThis method takes exactly what we put in the HTML and moves it directly into your CSS file.
Your CSS file (e.g., app.css or global.css):
CSS
@import "tailwindcss";
@layer components {
  .wrapper {
    @apply w-full mx-auto 
           md:border-x md:border-gray-200 md:max-w-[calc(100vw-2rem)] 
           min-[90rem]:max-w-[90rem]
           flex flex-col items-center gap-8 py-10;
  }
}
Option 2: The "Hybrid" Approach (Recommended)Since Tailwind v4 is designed to play incredibly nicely with modern, native CSS, you can mix @apply for the layout utilities (like flexbox and spacing) with the exact standard CSS media queries you originally wrote. This is often easier to read than a massive string of utilities.
Your CSS file:
CSS
@import "tailwindcss";
@layer components {
  .wrapper {
    /* Use Tailwind for the flex layout, centering, and spacing */
    @apply w-full mx-auto flex flex-col items-center gap-8 py-10;
  }
  /* Use standard CSS for your custom breakpoints and widths */
  @media (width >= 48rem) {
    .wrapper {
      @apply border-x border-gray-200; /* Still easy to apply borders here */
      max-width: calc(100vw - 2rem);
    }
  }
  @media (width >= 90rem) {
    .wrapper {
      max-width: 90rem;
    }
  }
}
(Note: Wrapping it in @layer components tells Tailwind to treat .wrapper like a component class, meaning it will be inserted at the right spot in the generated CSS to avoid specificity issues).
Your Cleaned-Up HTMLRegardless of which option you choose above, your HTML now becomes wonderfully simple:

You absolutely can use Astro's class:list with Tailwind CSS! In fact, it is the standard and most recommended way to handle dynamic classes in Astro.
However, there is a very specific Tailwind "Gotcha" you must understand when combining class:list with random numbers.
The Tailwind Compilation RuleTailwind CSS does not run in the browser; it runs during your build process. It scans your files looking for complete, unbroken text strings that match its class names.
Because of this, you cannot use JavaScript string interpolation to build an arbitrary Tailwind class on the fly.
❌ BAD: class:list={[\bg-[${randomX}%]`]}`
Why it fails: Tailwind sees the code literally as ${randomX}. It does not execute the math, so it never generates the CSS class for bg-[42%].
✅ GOOD: class:list={['bg-[42%]']}
Why it works: Tailwind sees the full string, understands it, and generates the CSS class.
How to use class:list for random backgroundsIf you want to use class:list for your random background position, you must create a pre-defined array of complete, valid Tailwind class strings, and then randomly select one of those strings.
Here is exactly how you would write that in your Astro component:
Extrait de code
---
// 1. Create a safe list of exact Tailwind classes you want to allow
const randomPositions = [
  'bg-top-left',
  'bg-center',
  'bg-bottom-right',
  'bg-[20%_50%]',  // Arbitrary values are totally fine here!
  'bg-[80%_10%]',
  'bg-[45%_90%]'
];
// 2. Pick one completely at random
const randomClass = randomPositions[Math.floor(Math.random() * randomPositions.length)];
---
<div 
  class:list={[
    "w-70 bg-[url(/fancy-background.png)]", // Your permanent classes
    randomClass                             // Your random class
  ]}
>
  </div>
When to use which method?Use the class:list method (above): If you only need a handful of specific, curated random positions (e.g., you want the background to jump between 5 or 6 cool spots).
Use the style or CSS Variable method (from my previous answer): If you want true randomness (e.g., literally any number between 1% and 100% so it is infinitely unique).
Does this pre-defined list approach work for the design you are trying to achieve, or do you need it to be 100% mathematically random?

To force an item in a CSS Grid to break to a new row, you don't use flex. Instead, you tell that specific grid item to span across all the columns, which naturally forces it onto its own line.
In your current code, you have nth-3:flex on the parent container. To make the 3rd child break to a new row, you can use Tailwind v4's child selector syntax combined with col-span-full (which tells it to take up the entire row).
Here is how you update your parent <div>:
Option 1: The Parent Target (Fixing your current approach)Replace nth-3:flex with [&>*:nth-child(3)]:col-span-full.
HTML
<div class="grid grid-cols-2 lg:grid-cols-3 auto-cols-max pt-20 gap-6.5 max-w-fit [&>*:nth-child(3)]:col-span-full">
  </div>
Why this works: [&>*:nth-child(3)] targets the 3rd direct child of the grid. col-span-full forces it to stretch across all columns (whether it's 2 or 3), pushing it to a brand new row.
Option 2: The Direct Target (Safer for conditional rendering)I noticed you are using conditional rendering (project.url_2 && ...). This is a potential trap!
If a project doesn't have a url_2, your Github link becomes the 1st child, and your Fork link becomes the 2nd child. The nth-child(3) rule would target nothing at all.
If you always want that project.fork link to sit on a new row no matter how many buttons came before it, the safest way is to apply col-span-full directly to that specific <a> tag:
Extrait de code
<div class="grid grid-cols-2 lg:grid-cols-3 auto-cols-max pt-20 gap-6.5 max-w-fit">
    {
        project.url_2 && 
        <a href={project.url_2} target="_blank" class="basic-button">
            <span data-twomi18n="find-project">Find project here</span>
        </a>
    }
    <a href={project.github_url} target="_blank" class="basic-button">
        <span data-twomi18n="see-on-github" class="pr-1">See on Github</span>
        </a>
    {
        project.fork &&
        <a href={project.fork} target="_blank" class="basic-button col-span-full">
            <span data-twomi18n="see-original" class="pr-1">See original project</span>
            </a>
    }
</div>
(Note: If you don't want the button to stretch visually across the whole screen, but just want it to drop to the left side of a new row, use col-start-1 instead of col-span-full!)

What kind of devices are between sm: and md: nowadays ?


========================================
GEMINI:
========================================
In the world of modern web design, the "no man's land" between Tailwind's sm: (640px) and md: (768px) breakpoints is actually quite crowded. This range primarily captures large smartphones in landscape mode and small tablets in portrait mode.
Here is a breakdown of what lives in that 128px gap:
1. Phablets and Large Phones (Landscape)Most "Plus," "Max," or "Ultra" phones fall squarely into this range when turned sideways.
iPhone 14/15/16 Pro Max: In landscape, these sit right around 667px - 730px (depending on the browser chrome).
Samsung Galaxy S24 Ultra: Landscape width is roughly 720px.
Google Pixel 8/9 Pro: Usually hits around 710px in landscape.
2. Small Tablets (Portrait)This is the "Mini" tablet zone.
iPad Mini: In portrait mode, the iPad Mini is exactly 744px or 768px wide (depending on the generation). Older versions sit exactly at the md: boundary, while newer ones sit just below it.
Amazon Fire HD 8: Portrait width is approximately 600px - 800px depending on resolution scaling.
3. Foldables (Unfolded)The new category of "Foldables" often has a "main" screen that sits right in this gap.
Samsung Galaxy Z Fold: When unfolded in portrait, it usually sits around 670px - 700px.
Google Pixel Fold: Similar behavior, often sitting around 700px wide.
Why this gap is a "Danger Zone" for LayoutsThe reason you often see "broken" layouts here is that designers often think in binaries: Mobile or Desktop.
At sm (640px): We assume it's a phone. We use single columns and big buttons.
At md (768px): We assume it's a tablet or laptop. We start using sidebars or multiple columns.
If you have a complex layout that snaps to 2 columns at md:, but is a single column at sm:, a user on a Galaxy Z Fold or an iPhone Pro Max (Landscape) sees a "stretched" mobile view. Everything looks giant and poorly spaced because the single column is covering 700+ pixels.
Best Practice Tip:If your content starts looking "stretched" or awkward before hitting the md: breakpoint, don't be afraid to use arbitrary breakpoints in Tailwind 4:
HTML
<div class="grid grid-cols-1 min-[700px]:grid-cols-2 md:grid-cols-3">
  ...
</div>
This ensures that those large-screen phone users get a slightly more "tablet-like" experience without you having to wait until the full 768px md: breakpoint.

To achieve this, you need a combination of setTimeout, clearTimeout, and mouse events (mouseenter and mouseleave).
The trick is to reset the 5-second timer every time the user's mouse enters the div. If the mouse leaves, you start the timer. If it comes back before 5 seconds are up, you cancel the timer so it doesn't hide.
Here is the exact JavaScript and HTML setup you need. Since you are using Tailwind, I'll use the hidden class to hide the element.
The HTMLHTML
<div id="hideable-div" class="w-64 h-64 bg-surface border border-border/20 p-5 transition-opacity duration-300">

The JavaScriptJavaScript
// 1. Target the div
const hideableDiv = document.getElementById('hideable-div');
// 2. Create a variable to hold the timer ID
let hideTimer;
// 3. Function to start the countdown
function startHideTimer() {
    // Always clear the previous timer first so they don't stack up!
    clearTimeout(hideTimer);
    // Set a new timer for 5000 milliseconds (5 seconds)
    hideTimer = setTimeout(() => {
        // Tailwind way to hide it
        hideableDiv.classList.add('hidden'); 
        // Alternative standard CSS way:
        // hideableDiv.style.display = 'none';
    }, 5000);
}
// 4. Function to stop the countdown
function cancelHideTimer() {
    clearTimeout(hideTimer);
}
// 5. Attach the event listeners
// When the mouse leaves the div -> start the 5-second countdown
hideableDiv.addEventListener('mouseleave', startHideTimer);
// When the mouse enters the div -> cancel the countdown
hideableDiv.addEventListener('mouseenter', cancelHideTimer);
// Optional: Call startHideTimer() right away if you want it to 
// disappear 5 seconds after page load (unless hovered).
startHideTimer();
How to use this in AstroIf you are putting this inside an Astro component, just drop the JavaScript inside a <script> tag at the bottom of your .astro file.

To give it a smooth, modern appearing (and disappearing) animation, we have to stop using the hidden class.
In CSS, the hidden class applies display: none. The browser cannot animate the transition between "not existing" and "existing," so it just instantly snaps on and off.
Instead, we will use Tailwind's opacity, transform (like scale or translate), and transition utilities. We'll also use pointer-events-none when it's invisible so it doesn't accidentally block you from clicking things underneath it.
Here is how you set up a beautiful "fade and slide up" animation.
1. The HTML (Tailwind Setup)We add transition-all, a duration, and an easing function to make it feel buttery smooth. We also start it in its "visible" state.
HTML
<div id="hideable-div" class="w-64 h-64 bg-surface border border-border/20 p-5 
  transition-all duration-500 ease-out
  opacity-100 translate-y-0 scale-100 pointer-events-auto
">
  <p class="text-content">Hover me to keep me alive. I will fade and slide away 5 seconds after you leave.</p>
</div>
2. The JavaScriptWe will update the JavaScript to swap out the visible classes for the hidden classes. We also need to make sure that if the user hovers over it while it's fading out, it immediately springs back to life!
JavaScript
const hideableDiv = document.getElementById('hideable-div');
let hideTimer;
// Group our classes to make the code super clean
const visibleClasses = ['opacity-100', 'translate-y-0', 'scale-100', 'pointer-events-auto'];
const hiddenClasses = ['opacity-0', 'translate-y-4', 'scale-95', 'pointer-events-none'];
function hideElement() {
    // Remove visible state, add hidden state
    hideableDiv.classList.remove(...visibleClasses);
    hideableDiv.classList.add(...hiddenClasses);
}
function showElement() {
    // Remove hidden state, add visible state
    hideableDiv.classList.remove(...hiddenClasses);
    hideableDiv.classList.add(...visibleClasses);
}
function startHideTimer() {
    clearTimeout(hideTimer);
    // Set timer to hide after 5 seconds
    hideTimer = setTimeout(() => {
        hideElement();
    }, 5000);
}
function cancelHideTimer() {
    clearTimeout(hideTimer);
    // If the user's mouse enters the div, instantly show it again 
    // (in case it was already in the middle of fading out)
    showElement();
}
// Attach the listeners
hideableDiv.addEventListener('mouseleave', startHideTimer);
hideableDiv.addEventListener('mouseenter', cancelHideTimer);
// Start the countdown as soon as the page loads
startHideTimer();
How the animation works:opacity-0 to opacity-100: Fades the element in and out.
translate-y-4 to translate-y-0: Makes it slide up slightly when appearing, and sink down when disappearing.
scale-95 to scale-100: Gives it a subtle "pop" effect by growing slightly as it enters.
Because of transition-all duration-500 ease-out, the browser automatically tweens all of these values smoothly over half a second!
